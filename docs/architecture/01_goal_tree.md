# Дерево цілей — IPSO Detector

> Розділ 2.1 дипломної роботи: Формування вимог та постановка задачі

---

## Генеральна мета

**Розробити розподілену інформаційну систему для автоматичного виявлення ознак російських інформаційно-психологічних операцій (ІПСО) у текстовому контенті з використанням методів штучного інтелекту.**

### А1. Точність аналізу (якість виявлення ІПСО)

Забезпечити високу якість класифікації текстів з мінімізацією хибних спрацювань.

- **А1.1** Забезпечити розпізнавання ≥7 типів пропагандистських наративів (деморалізація, підрив довіри, хибна рівність, паніка, перебільшення втрат, «Захід зраджує», корупційні звинувачення)
- **А1.2** Забезпечити виявлення ≥6 маніпулятивних риторичних технік (whataboutism, хибний авторитет, нагнітання страху, деморалізація, хибна рівність, cherry-picking)
- **А1.3** Забезпечити порівняння тексту з базою відомих ІПСО-наративів через семантичну подібність (cosine similarity ≥ 0.55)
- **А1.4** Забезпечити підтримку багатомовності (українська, російська, англійська) з автоматичним перекладом для уніфікації аналізу
- **А1.5** Забезпечити ансамблевий скоринг (зважена комбінація 3 модулів аналізу) для формування підсумкового вердикту

### А2. Доступність та швидкість (продуктивність системи)

Забезпечити швидку та надійну роботу системи для кінцевих користувачів.

- **А2.1** Забезпечити час відповіді API Gateway ≤ 300ms (для прийняття запиту та підтвердження)
- **А2.2** Забезпечити повний час аналізу тексту ≤ 10 секунд (з урахуванням виклику зовнішнього AI API)
- **А2.3** Забезпечити доступність системи ≥ 99% (автоматичний перезапуск через systemd)
- **А2.4** Забезпечити асинхронну обробку запитів через чергу повідомлень (Celery + Redis) для уникнення блокування

### А3. Масштабованість (розподіленість системи)

Забезпечити можливість горизонтального масштабування компонентів системи.

- **А3.1** Розділити систему на логічно незалежні сервіси (API Gateway, Bot Service, Analyzer Workers, Data Layer)
- **А3.2** Забезпечити можливість запуску декількох Analyzer Workers паралельно
- **А3.3** Забезпечити комунікацію між сервісами через брокер повідомлень (Redis)
- **А3.4** Забезпечити незалежне розгортання та оновлення кожного сервісу
- **А3.5** Забезпечити відмовостійкість — автоматичне перевидання задачі при збої воркера

### А4. Зручність використання (UX)

Забезпечити простий та інтуїтивний інтерфейс для різних категорій користувачів.

- **А4.1** Telegram-бот: аналіз за одне пересилання/надсилання повідомлення, відповідь з візуальним вердиктом
- **А4.2** Веб-інтерфейс: форма для вставки тексту, візуалізація результатів, історія аналізів
- **А4.3** REST API: стандартний JSON API для інтеграції із зовнішніми системами
- **А4.4** Зворотний зв'язок: можливість оцінити результат аналізу (👍/👎) для майбутнього вдосконалення
- **А4.5** Адмін-панель: перегляд статистики, управління базою наративів, моніторинг якості

## Рівень 2 — Критерії якості (метрики)

| Критерій | Метрика | Цільове значення |
|----------|---------|------------------|
| Точність класифікації | Precision на тестовому наборі 20+ текстів | ≥ 75% |
| Повнота виявлення | Recall ІПСО-текстів | ≥ 70% |
| Швидкість відповіді | P95 latency повного аналізу | ≤ 10с |
| Швидкість прийому | Час підтвердження отримання запиту | ≤ 300ms |
| Доступність | Uptime | ≥ 99% |
| Масштабованість | Горизонтальне масштабування воркерів | 1→N без зміни коду |
| Задоволеність | Частка позитивних відгуків | ≥ 60% |
| Багатомовність | Підтримувані мови | uk, ru, en |

## Альтернативні варіанти побудови системи

Для вибору оптимального варіанту побудови системи розглянуто 3 альтернативи:

### Альтернатива 1: Монолітна архітектура

Єдиний Django-додаток, який обробляє все синхронно: Telegram webhook, AI-аналіз, веб-інтерфейс, API.

| Критерій | Оцінка |
|----------|--------|
| Простота розробки | Висока — один процес, один деплой |
| Масштабованість | Низька — вертикальне масштабування (більше RAM/CPU) |
| Відмовостійкість | Низька — збій AI-модуля = збій всієї системи |
| Час відповіді | Високий — синхронний аналіз блокує webhook |
| Відповідність темі «Розподілені ІС» | Не відповідає |

### Альтернатива 2: Сервісно-орієнтована архітектура (SOA) з чергою повідомлень

Django API Gateway + окремі Celery Workers для аналізу + Redis як брокер + окремий бот-процес. Компоненти логічно розділені, комунікують через Redis та внутрішній REST.

| Критерій | Оцінка |
|----------|--------|
| Простота розробки | Середня — одна кодова база, декілька процесів |
| Масштабованість | Висока — горизонтальне масштабування воркерів |
| Відмовостійкість | Висока — воркери незалежні, retry при збоях |
| Час відповіді | Низький — async, Gateway відповідає миттєво |
| Відповідність темі «Розподілені ІС» | Повна відповідність |

### Альтернатива 3: Мікросервісна архітектура

Кожен модуль (translator, rhetoric, narrative, similarity) — окремий HTTP-сервіс у Docker-контейнері з власним API.

| Критерій | Оцінка |
|----------|--------|
| Простота розробки | Низька — складна оркестрація, Docker Compose, мережа |
| Масштабованість | Дуже висока — кожен сервіс масштабується окремо |
| Відмовостійкість | Дуже висока — повна ізоляція сервісів |
| Час відповіді | Середній — overhead мережевих викликів між сервісами |
| Відповідність темі «Розподілені ІС» | Повна, але надлишкова для MVP |

### Вибір: Альтернатива 2 — SOA з чергою повідомлень

**Обґрунтування:** Альтернатива 2 забезпечує оптимальний баланс між складністю розробки (одна кодова база Django) та рівнем розподіленості (логічно незалежні процеси, горизонтальне масштабування, асинхронна комунікація через Redis). Мікросервіси (Альтернатива 3) є надлишковими для проєкту з командою одного розробника та дедлайном у 10 тижнів. Моноліт (Альтернатива 1) не відповідає вимогам теми «Розподілені ІС».

> **Примітка для диплому:** У розділі 2.1 цей вибір оформлюється через метод аналітичної ієрархії (AHP) з попарним порівнянням критеріїв та альтернатив.
