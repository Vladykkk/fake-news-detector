# Вимоги до системи IPSO Detector

> Розділ 2.1 дипломної роботи: Формування вимог та постановка задачі

---

## Об'єкт дослідження

Процес автоматичного виявлення інформаційно-психологічних операцій (ІПСО) у текстовому контенті з використанням розподілених інформаційних систем та методів штучного інтелекту.

## Предмет дослідження

Методи та засоби побудови розподіленої інформаційної системи для класифікації текстових повідомлень за ознаками пропагандистських наративів із застосуванням NLP, rule-based аналізу та семантичного порівняння.

---

## Функціональні вимоги (FR)

### FR1. Прийом та обробка вхідних даних

- **FR1.1** Система повинна приймати текстові повідомлення через Telegram-бот (пряме введення та пересилання)
- **FR1.2** Система повинна приймати текст через REST API (POST /api/analyze/)
- **FR1.3** Система повинна приймати текст через веб-інтерфейс (форма введення)
- **FR1.4** Система повинна автоматично визначати мову тексту (uk, ru, en та інші)
- **FR1.5** Система повинна автоматично перекладати текст на англійську для уніфікації аналізу
- **FR1.6** Система повинна відхиляти тексти коротші за 30 символів з повідомленням помилки

### FR2. Аналіз контенту

- **FR2.1** Система повинна класифікувати текст за 7+ категоріями пропагандистських наративів із використанням LLM через OpenRouter API
- **FR2.2** Система повинна виявляти маніпулятивні риторичні техніки за допомогою rule-based аналізу (regex-патерни) у 6 категоріях
- **FR2.3** Система повинна порівнювати текст з базою відомих ІПСО-наративів через cosine similarity на основі sentence embeddings
- **FR2.4** Система повинна обчислювати підсумковий бал ІПСО як зважену комбінацію трьох модулів аналізу: narrative (0.40), rhetoric (0.30), similarity (0.30)
- **FR2.5** Система повинна класифікувати результат за трьома рівнями: «Безпечно» (< 0.35), «Підозрілий» (0.35–0.70), «ІПСО» (> 0.70)

### FR3. Представлення результатів

- **FR3.1** Telegram-бот повинен відправляти форматований результат аналізу з: вердиктом, відсотком, деталями по кожному модулю, виявленими наративами та техніками
- **FR3.2** Веб-інтерфейс повинен відображати результат у вигляді інтерактивної картки з кольоровою індикацією рівня загрози
- **FR3.3** REST API повинен повертати повний результат аналізу у форматі JSON
- **FR3.4** Система повинна зберігати всі результати аналізу в базі даних

### FR4. Зворотній зв'язок та управління

- **FR4.1** Telegram-бот повинен надавати inline-кнопки для оцінки результату (👍 Правильно / 👎 Помилка)
- **FR4.2** Адмін-панель повинна відображати статистику аналізів (загальна кількість, розподіл за вердиктами)
- **FR4.3** Адмін-панель повинна дозволяти управляти базою відомих ІПСО-наративів (CRUD)
- **FR4.4** Система повинна зберігати зворотний зв'язок користувачів та прив'язувати його до конкретних результатів аналізу
- **FR4.5** REST API повинен надавати ендпоінт для перегляду історії аналізів (GET /api/history/)

### FR5. Команди бота

- **FR5.1** /start — привітання та опис можливостей бота
- **FR5.2** /help — детальна довідка з описом функціональності
- **FR5.3** /stats — статистика роботи бота (кількість аналізів, розподіл вердиктів)

## Нефункціональні вимоги (NFR)

### NFR1. Продуктивність

- **NFR1.1** Час прийому запиту (API Gateway response): ≤ 300ms
- **NFR1.2** Час повного аналізу тексту: ≤ 10 секунд (P95)
- **NFR1.3** Пропускна здатність: ≥ 50 аналізів/хвилину при 2 Celery workers
- **NFR1.4** Розмір тексту для аналізу: від 30 до 10 000 символів

### NFR2. Надійність та відмовостійкість

- **NFR2.1** Доступність системи: ≥ 99% (uptime)
- **NFR2.2** Автоматичний перезапуск сервісів через systemd (restart=on-failure)
- **NFR2.3** Автоматичне перевидання задач Celery при збої воркера (max_retries=3)
- **NFR2.4** Telegram webhook завжди повертає 200 OK для уникнення retry-штормів

### NFR3. Масштабованість

- **NFR3.1** Горизонтальне масштабування Analyzer Workers (1→N) без зміни коду
- **NFR3.2** Логічне розділення сервісів: API Gateway, Bot Service, Analyzer Workers, Data Layer
- **NFR3.3** Асинхронна комунікація між сервісами через Redis (message broker)
- **NFR3.4** Stateless Analyzer Workers (стан зберігається в PostgreSQL/Redis)

### NFR4. Безпека

- **NFR4.1** HTTPS (TLS 1.2+) для всіх зовнішніх з'єднань
- **NFR4.2** Валідація Telegram webhook через secret_token
- **NFR4.3** Зберігання секретів у .env файлі (не в коді)
- **NFR4.4** CSRF-захист для веб-інтерфейсу
- **NFR4.5** Rate limiting на API ендпоінтах (захист від зловживань)

### NFR5. Розгортання та підтримка

- **NFR5.1** Розгортання на DigitalOcean Ubuntu 24.04 droplet
- **NFR5.2** Reverse proxy: nginx з Let's Encrypt SSL
- **NFR5.3** Управління процесами: systemd (gunicorn, celery, bot)
- **NFR5.4** Логування всіх подій аналізу (рівень INFO) та помилок (рівень ERROR)
- **NFR5.5** Можливість локального запуску в dev-режимі без Celery (синхронний аналіз)

### NFR6. Якість коду

- **NFR6.1** Мова програмування: Python 3.11+
- **NFR6.2** Фреймворк: Django 4.2 LTS
- **NFR6.3** API: Django REST Framework
- **NFR6.4** Покриття тестами ключових модулів (analyzer pipeline)
- **NFR6.5** Документування коду (docstrings для публічних функцій)
