# Основні засади розподілених інформаційних систем та технологій

> Розділ 1.1 дипломної роботи

---

## 1. Поняття розподіленої інформаційної системи

Розподілена інформаційна система (РІС) — це сукупність автономних обчислювальних вузлів, з'єднаних комп'ютерною мережею, які взаємодіють між собою для досягнення спільної мети та представляються кінцевому користувачу як єдина цілісна система [1]. На відміну від централізованих систем, де всі обчислення та зберігання даних зосереджені на одному сервері, розподілені системи розподіляють навантаження між декількома вузлами, що забезпечує масштабованість, відмовостійкість та ефективне використання ресурсів.

Ендрю Таненбаум визначає ключову властивість розподіленої системи: «Розподілена система — це набір незалежних комп'ютерів, які представляються користувачам як єдина узгоджена система» [2]. Це означає, що внутрішня складність розподілу прихована від користувача за єдиним інтерфейсом.

Основними характеристиками розподілених інформаційних систем є:

**Прозорість (transparency)** — здатність системи приховувати від користувача та розробника деталі розподілу. Виділяють прозорість доступу (єдиний спосіб доступу до локальних і віддалених ресурсів), прозорість розташування (користувач не знає фізичного розміщення ресурсу), прозорість переміщення (ресурси можуть переміщатись без впливу на роботу), прозорість реплікації (копії даних підтримуються без відома користувача) та прозорість відмов (система продовжує працювати при збоях окремих компонентів) [2].

**Відкритість (openness)** — система будується на стандартних протоколах та інтерфейсах, що дозволяє інтеграцію компонентів від різних розробників. У контексті веб-систем це означає використання стандартних протоколів HTTP/HTTPS, форматів JSON/XML та архітектурних стилів REST [3].

**Масштабованість (scalability)** — здатність системи збільшувати продуктивність при додаванні нових ресурсів. Розрізняють вертикальне масштабування (збільшення потужності одного вузла) та горизонтальне масштабування (додавання нових вузлів). Горизонтальне масштабування є ключовою перевагою розподілених систем [4].

**Відмовостійкість (fault tolerance)** — здатність системи продовжувати коректну роботу при виході з ладу окремих компонентів. Досягається через надлишковість (redundancy), реплікацію даних та механізми автоматичного відновлення [5].

## 2. Архітектурні моделі розподілених систем

### 2.1. Клієнт-серверна архітектура

Найпоширеніша модель розподілених систем, де компоненти поділяються на клієнти (ініціатори запитів) та сервери (провайдери послуг). У багаторівневій (n-tier) архітектурі виділяються рівні представлення, бізнес-логіки та даних, кожен з яких може бути розміщений на окремому вузлі [1].

Типова трирівнева веб-архітектура складається з: веб-сервера (nginx, Apache) на рівні представлення, сервера додатків (Django, Flask, Node.js) на рівні бізнес-логіки та сервера бази даних (PostgreSQL, MySQL) на рівні даних.

### 2.2. Сервісно-орієнтована архітектура (SOA)

SOA (Service-Oriented Architecture) — архітектурний підхід, у якому система будується як набір слабо зв'язаних сервісів, що взаємодіють через визначені інтерфейси [6]. Кожен сервіс є автономною одиницею функціональності, яка може бути незалежно розгорнута, масштабована та оновлена.

Ключові принципи SOA: слабке зв'язування (loose coupling) — сервіси мінімально залежать один від одного; повторне використання (reusability) — сервіси можуть бути використані різними клієнтами; автономність (autonomy) — кожен сервіс контролює свою логіку та ресурси; відсутність стану (statelessness) — сервіси не зберігають стан між викликами [6].

У контексті сучасних веб-додатків SOA реалізується через REST API, черги повідомлень (message queues) та шину подій (event bus). Це забезпечує асинхронну комунікацію між компонентами та можливість горизонтального масштабування.

### 2.3. Мікросервісна архітектура

Мікросервісна архітектура — еволюція SOA, де кожен сервіс є максимально малим, відповідає за одну бізнес-функцію та має власне сховище даних [7]. Мікросервіси зазвичай розгортаються у контейнерах (Docker) та оркеструються через платформи типу Kubernetes.

Переваги мікросервісів: незалежне розгортання, вибір технологічного стеку для кожного сервісу, ізоляція відмов. Недоліки: складність оркестрації, overhead мережевої комунікації, складність налагодження розподілених транзакцій, необхідність DevOps-інфраструктури [7].

### 2.4. Архітектура з чергою повідомлень (Message Queue Architecture)

Шаблон архітектури, де компоненти комунікують через посередника — чергу повідомлень (message broker). Виробник (producer) публікує повідомлення у чергу, споживач (consumer/worker) отримує та обробляє їх асинхронно [8].

Ключові характеристики: асинхронність (виробник не чекає відповіді), розвантаження (тривалі задачі не блокують основний процес), надійність (повідомлення зберігаються в черзі до успішної обробки), масштабованість (можна додавати нових споживачів без зміни виробника).

Популярні реалізації: RabbitMQ (AMQP), Apache Kafka (event streaming), Redis (легковаговий брокер). У Python-екосистемі Celery є стандартним рішенням для розподілених задач з підтримкою Redis та RabbitMQ як брокерів [9].

## 3. Теорема CAP та її наслідки

Теорема CAP (Brewer's theorem), сформульована Еріком Брюером у 2000 році та доведена Сетом Гілбертом і Ненсі Лінч у 2002 році, стверджує, що розподілена система може одночасно забезпечити лише дві з трьох властивостей [10]:

**Consistency (узгодженість)** — усі вузли бачать одні й ті ж дані в один і той самий час.

**Availability (доступність)** — кожен запит отримує відповідь (успішну або помилку), навіть при збоях окремих вузлів.

**Partition tolerance (стійкість до розділення)** — система продовжує працювати при втраті з'єднання між вузлами.

Оскільки мережеві розділення (partitions) неминучі у реальних розподілених системах, практичний вибір стоїть між CP-системами (узгодженість + стійкість, наприклад PostgreSQL з реплікацією) та AP-системами (доступність + стійкість, наприклад Cassandra, DynamoDB) [10].

Для інформаційної системи виявлення ІПСО оптимальним є CP-підхід: узгодженість результатів аналізу критичніша за абсолютну доступність, оскільки хибний результат (через неузгодженість) небезпечніший за тимчасову недоступність.

## 4. Асинхронна обробка у розподілених системах

Асинхронна модель обробки є фундаментальною для розподілених систем, де час відповіді зовнішніх сервісів є непередбачуваним [11]. У контексті веб-додатків виділяють два підходи:

**Синхронна обробка (request-response)** — клієнт надсилає запит та блокується до отримання відповіді. Простий у реалізації, але непридатний для тривалих операцій (AI-аналіз, виклик зовнішніх API), оскільки блокує веб-сервер та може призвести до таймаутів.

**Асинхронна обробка (task queue)** — клієнт надсилає запит, отримує миттєве підтвердження, а фактична обробка виконується у фоновому процесі (worker). Результат доставляється через callback, webhook або polling [9].

Патерн «виробник-споживач» (producer-consumer) у поєднанні з чергою повідомлень забезпечує: розв'язання (decoupling) між прийомом запиту та його обробкою; буферизацію при піковому навантаженні; можливість повторної обробки при збоях (retry); горизонтальне масштабування шляхом додавання споживачів.

## 5. Інтеграція штучного інтелекту в розподілені системи

Сучасні системи обробки природної мови (NLP) вимагають значних обчислювальних ресурсів, що робить їх інтеграцію у розподілені системи особливо актуальною [12].

### 5.1. Моделі розгортання AI

**Локальний inference** — модель завантажується та виконується на тому ж сервері, що й додаток. Забезпечує мінімальну затримку, але вимагає значних ресурсів (GPU/CPU, RAM). Приклад: sentence-transformers, spaCy.

**Віддалений inference через API** — модель розгортається як окремий сервіс (AI-as-a-Service), додаток надсилає запити через REST API. Зменшує вимоги до локальних ресурсів, але додає мережеву затримку та залежність від зовнішнього сервісу. Приклади: OpenAI API, HuggingFace Inference API, OpenRouter [13].

**Гібридний підхід** — легкі моделі (embeddings, rule-based аналіз) виконуються локально, а важкі (LLM для класифікації) — через зовнішній API. Оптимальний баланс між продуктивністю, вартістю та вимогами до ресурсів.

### 5.2. Ансамблевий підхід до класифікації

Ансамблеві методи (ensemble methods) поєднують результати декількох моделей або алгоритмів для підвищення якості класифікації [14]. У задачі виявлення дезінформації ансамблевий підхід є особливо ефективним, оскільки різні методи виявляють різні аспекти маніпулятивного контенту:

Зважений ансамбль (weighted ensemble) — кожна модель отримує вагу відповідно до її внеску у фінальне рішення. Фінальний скор обчислюється як зважена сума: S = w₁·s₁ + w₂·s₂ + ... + wₙ·sₙ, де wᵢ — вага i-того модуля, sᵢ — скор i-того модуля [14].

### 5.3. Методи NLP для виявлення дезінформації

Сучасні дослідження у сфері автоматичного виявлення фейкових новин використовують декілька підходів [15]:

**Rule-based аналіз** — використання регулярних виразів та словників для виявлення відомих маніпулятивних патернів (whataboutism, хибний авторитет, нагнітання страху). Швидкий та інтерпретований, але обмежений відомими шаблонами.

**Класифікація на основі LLM** — великі мовні моделі (BERT, GPT, LLaMA) здатні класифікувати текст за наративними категоріями без додаткового навчання через zero-shot або few-shot prompting. Найновіші дослідження (2025) показують, що BERT досягає 98% accuracy на стандартних бенчмарках виявлення фейкових новин [15].

**Семантичне порівняння (semantic similarity)** — використання sentence embeddings (sentence-transformers) для обчислення косинусної подібності між аналізованим текстом та базою відомих пропагандистських наративів. Ефективний для виявлення перефразованих варіантів відомих ІПСО-кампаній [16].

## 6. Технології комунікації у розподілених веб-системах

### 6.1. REST (Representational State Transfer)

REST — архітектурний стиль для побудови розподілених гіпермедійних систем, запропонований Роєм Філдінгом у 2000 році [3]. Ключові обмеження REST: клієнт-серверна архітектура, відсутність стану (stateless), кешованість (cacheable), одноманітний інтерфейс (uniform interface), багаторівнева система (layered system).

У контексті сучасних веб-API стандартом є використання HTTP-методів (GET, POST, PUT, DELETE) з обміном даними у форматі JSON. Django REST Framework (DRF) забезпечує декларативне створення REST API з серіалізацією, валідацією та документуванням.

### 6.2. Webhook-архітектура

Webhook — це механізм зворотного виклику через HTTP, де сервер A (наприклад, Telegram) надсилає HTTP POST запит до сервера B (наш API Gateway) при настанні певної події [17]. На відміну від polling (періодичне опитування), webhook забезпечує доставку подій у реальному часі без зайвого навантаження.

Telegram Bot API підтримує два режими роботи: long polling (бот періодично запитує оновлення) та webhook (Telegram надсилає оновлення на вказаний URL). Webhook є рекомендованим для production-систем, оскільки забезпечує нижчу затримку та менше навантаження [17].

### 6.3. Reverse Proxy

Reverse proxy — серверний компонент, який приймає запити від клієнтів та перенаправляє їх до внутрішніх серверів [18]. nginx є стандартним рішенням, яке забезпечує: TLS-термінацію (SSL/HTTPS), балансування навантаження, кешування статичного контенту, захист від DDoS-атак, маршрутизацію запитів до різних бекенд-сервісів за URL-шаблоном.

## 7. Релевантність розподілених технологій для задачі виявлення ІПСО

Задача автоматичного виявлення інформаційно-психологічних операцій є типовим прикладом проблеми, яка потребує розподіленого підходу з наступних причин:

1. **Тривалість AI-аналізу.** Виклик зовнішнього LLM API займає 2–8 секунд, що неприпустимо для синхронної обробки webhook-запитів (Telegram вимагає відповідь за 60 секунд).

2. **Множинність джерел вхідних даних.** Система приймає запити з трьох каналів (Telegram, веб, API), що потребує єдиного API Gateway з маршрутизацією.

3. **Незалежність модулів аналізу.** Три модулі аналізу (rhetoric, narrative, similarity) логічно незалежні та можуть виконуватись паралельно, що відповідає шаблону fork-join у розподілених системах.

4. **Залежність від зовнішніх сервісів.** Система залежить від OpenRouter API та Google Translate, збої яких не повинні впливати на доступність основного сервісу (graceful degradation).

5. **Масштабованість навантаження.** При вірусному поширенні пропагандистського контенту кількість запитів може зростати стрибкоподібно, що потребує горизонтального масштабування worker-процесів.
