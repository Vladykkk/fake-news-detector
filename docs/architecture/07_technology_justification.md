# Вибір та обґрунтування програмних і технічних засобів

> Розділ 3.2 дипломної роботи

---

## 1. Мова програмування: Python 3.11+

### Розглянуті альтернативи

| Мова | Переваги | Недоліки |
|------|----------|----------|
| **Python 3.11** | Найбагатша екосистема для AI/ML та NLP; Django — зрілий веб-фреймворк; Celery — стандартний task queue; sentence-transformers, langdetect, deep-translator — готові бібліотеки | Повільніший за компільовані мови; GIL обмежує паралелізм у рамках одного процесу |
| JavaScript (Node.js) | Висока швидкість I/O; єдина мова для frontend та backend | Бідна AI/ML екосистема; відсутність зрілих аналогів Celery; TensorFlow.js суттєво обмежений |
| Go | Висока продуктивність; нативна підтримка конкурентності (goroutines) | Відсутність AI/ML екосистеми; немає аналогів Django ORM та DRF; складніша розробка CRUD |
| Java (Spring Boot) | Enterprise-рівень; хороша екосистема NLP (Stanford NLP, OpenNLP) | Більший boilerplate; повільніший цикл розробки; важчий деплой |

### Обґрунтування вибору

Python обрано як основну мову програмування з наступних причин:

1. **AI/ML екосистема.** Python є де-факто стандартом для розробки систем штучного інтелекту. Бібліотеки sentence-transformers (для semantic similarity), transformers (HuggingFace), langdetect та deep-translator мають саме Python-реалізації як основні.

2. **Celery.** Єдина промислово-зріла бібліотека для розподіленої обробки задач із підтримкою retry, scheduling та моніторингу. Celery написаний на Python і не має повноцінних аналогів в інших мовах.

3. **Django.** Зрілий веб-фреймворк (версія 4.2 LTS) з вбудованою ORM, admin-панеллю, системою міграцій та великою спільнотою. Django REST Framework забезпечує швидке створення REST API з серіалізацією, валідацією та документуванням.

4. **python-telegram-bot.** Зрілий async-фреймворк для побудови Telegram-ботів з підтримкою webhook-режиму, inline-кнопок та callback-обробників.

5. **Швидкість розробки.** Для проєкту з одним розробником та обмеженим терміном (10 тижнів) Python забезпечує найвищу продуктивність завдяки мінімальному boilerplate та великій кількості готових бібліотек.

Обмеження Python щодо продуктивності (GIL) компенсується архітектурним рішенням: важкі обчислення виконуються в окремих Celery worker-процесах, а не потоках, що обходить GIL. Кожен worker — це окремий Python-процес з власним інтерпретатором.

## 2. Веб-фреймворк: Django 4.2 LTS + Django REST Framework

### Розглянуті альтернативи

| Фреймворк | Переваги | Недоліки |
|-----------|----------|----------|
| **Django 4.2 + DRF** | Вбудована ORM + міграції + admin; DRF для REST API; великий набір middleware (CORS, CSRF, auth); LTS-підтримка до квітня 2026 | Монолітна архітектура "за замовчуванням"; синхронний за замовчуванням (async підтримка обмежена) |
| FastAPI | Async-native; автоматична OpenAPI документація; висока швидкість | Відсутність вбудованої ORM та admin; потребує SQLAlchemy + Alembic окремо; менша спільнота |
| Flask | Мінімалізм; гнучкість | Відсутність вбудованих компонентів (ORM, admin, форми); потребує значної кількості extensions |

### Обґрунтування вибору

Django 4.2 LTS обрано як основний веб-фреймворк:

1. **"Batteries included".** Django надає з коробки: ORM з підтримкою PostgreSQL (включаючи JSONField для зберігання результатів аналізу), систему міграцій (автоматичне управління схемою БД), admin-панель (готовий інтерфейс для перегляду результатів та управління наративами), middleware для безпеки (CSRF, clickjacking, XSS).

2. **Django REST Framework.** DRF надає серіалізатори (автоматичне перетворення Django-моделей у JSON), ViewSets та APIViews (декларативне створення API-ендпоінтів), валідацію вхідних даних, throttling (rate limiting).

3. **Admin-панель.** Django admin забезпечує повноцінний інтерфейс адміністратора без додаткового коду: перегляд та фільтрація результатів аналізу, CRUD-операції для бази відомих наративів, перегляд статистики зворотного зв'язку. Це критично для дипломної роботи, де потрібно продемонструвати функціонуючий інтерфейс управління.

4. **LTS-підтримка.** Django 4.2 має підтримку до квітня 2026, що забезпечує стабільність на період розробки та захисту.

FastAPI є сильним кандидатом для нових async-проєктів, але для даного проєкту Django забезпечує значно швидший час розробки завдяки вбудованій admin-панелі та ORM. Асинхронність обробки забезпечується Celery, а не фреймворком.

## 3. База даних: PostgreSQL 16

### Розглянуті альтернативи

| СКБД | Переваги | Недоліки |
|------|----------|----------|
| **PostgreSQL 16** | Повна підтримка JSONField (jsonb); повнотекстовий пошук; надійність та ACID; розширення (pgvector для embeddings); найкраща інтеграція з Django | Більше ресурсів ніж SQLite; потребує окремого серверу |
| SQLite | Вбудована; zero-config; підходить для dev | Не підтримує конкурентний запис; обмежена підтримка JSON; не рекомендується для production |
| MongoDB | Нативна підтримка JSON; гнучка схема | Відсутність транзакцій ACID (до v4.0 обмежено); слабша інтеграція з Django; відсутність стандартних міграцій |
| MySQL | Широке поширення; хороша продуктивність | Обмежена підтримка JSON (порівняно з PostgreSQL jsonb); слабша підтримка в Django для складних запитів |

### Обґрунтування вибору

PostgreSQL обрано як основну СКБД:

1. **JSONField (jsonb).** Результати аналізу містять складні структуровані дані: detected_narratives (масив об'єктів з label та confidence), detected_rhetoric (масив з type та snippet), similar_narratives (масив з title та similarity), embedding (вектор 384 вимірів). PostgreSQL jsonb забезпечує ефективне зберігання, індексацію та запити до JSON-даних, що дозволяє використовувати реляційну модель для основних полів та JSON для варіативних структур.

2. **ACID-транзакції.** Гарантують цілісність даних при конкурентному записі від декількох Celery workers. Кожен worker може одночасно зберігати результати аналізу без ризику втрати або пошкодження даних.

3. **Django ORM.** PostgreSQL є рекомендованою СКБД для Django, з найповнішою підтримкою: ArrayField, JSONField, HStoreField, повнотекстовий пошук, а також агрегатні функції для JSON.

4. **Перспектива масштабування.** PostgreSQL підтримує реплікацію (streaming replication), партиціонування таблиць та розширення pgvector для зберігання та пошуку векторних embeddings — що може бути використано у майбутніх версіях для прискорення similarity search.

## 4. Черга повідомлень: Celery 5 + Redis 7

### Розглянуті альтернативи

| Рішення | Переваги | Недоліки |
|---------|----------|----------|
| **Celery 5 + Redis** | Стандарт для Python; retry, scheduling, monitoring; Redis як легковаговий брокер + кеш; мінімальне використання пам'яті | Celery 5 вимагає Python 3.8+; Redis — in-memory (потребує моніторингу RAM) |
| Celery + RabbitMQ | RabbitMQ — повноцінний AMQP-брокер; надійніше збереження повідомлень | RabbitMQ потребує більше RAM та конфігурації; надлишковий для проєкту з <100 задач/хв |
| Django Q | Простіша інтеграція з Django; використовує Django ORM як брокер | Менша спільнота; відсутність вбудованого retry; менше документації |
| Huey | Легковагова альтернатива Celery; Redis backend | Менша функціональність; немає вбудованого моніторингу (Flower) |

### Обґрунтування вибору

Celery 5 з Redis як брокером обрано як рішення для асинхронної обробки:

1. **Celery — стандарт.** Celery є найпоширенішим рішенням для розподілених задач у Python-екосистемі з документацією, яка охоплює всі аспекти production-деплою. Забезпечує: автоматичні retry при збоях (max_retries=3), серіалізацію задач у JSON, розподіл задач між декількома workers, моніторинг через Flower.

2. **Redis — подвійна роль.** Redis використовується одночасно як message broker для Celery та як cache для часто використовуваних даних (наприклад, embeddings відомих наративів). Це зменшує кількість залежностей інфраструктури з 3 (Django + брокер + кеш) до 2 (Django + Redis).

3. **Горизонтальне масштабування.** Для збільшення пропускної здатності достатньо запустити додаткові Celery workers командою: `celery -A config worker --concurrency=N`. Нові workers автоматично підключаються до Redis та починають обробляти задачі з черги.

4. **Мінімальне споживання ресурсів.** Redis споживає ~10–30MB RAM для нашого навантаження (<100 задач/хвилину), що критично для обмеженого бюджету DigitalOcean droplet.

## 5. AI-аналіз: OpenRouter API + sentence-transformers

### Розглянуті альтернативи для класифікації наративів

| Підхід | Переваги | Недоліки |
|--------|----------|----------|
| **OpenRouter API** | Доступ до десятків LLM (Claude, GPT, Llama, Qwen); pay-per-use; не потребує локальних ресурсів; гнучкість вибору моделі | Залежність від зовнішнього API; затримка 2–8с; вартість запитів |
| Локальний BERT (facebook/bart-large-mnli) | Без залежності від зовнішнього API; передбачувана затримка | Потребує 2–4GB RAM; повільний на CPU (5–15с); обмежена якість zero-shot |
| HuggingFace Inference API | Безкоштовний tier; зручний REST API | Rate limits; непередбачувана доступність; cold start 20–60с |
| Fine-tuned модель | Найвища якість для вузької задачі | Потребує великого датасету для навчання; значний час розробки; GPU для навчання |

### Обґрунтування вибору

**OpenRouter API** обрано для класифікації наративів (модуль narrative), а **sentence-transformers (paraphrase-multilingual-MiniLM-L12-v2)** — для локального обчислення embeddings (модуль similarity):

1. **OpenRouter — гнучкість.** OpenRouter надає єдиний API для доступу до десятків LLM-провайдерів. Якщо одна модель стає недоступною або дорогою, можна переключитись на іншу без зміни коду. Для нашої задачі оптимальні: Claude Haiku (швидкий, дешевий, добре розуміє контекст) або GPT-4o-mini (хороша якість за низьку ціну).

2. **Економія ресурсів.** LLM inference потребує 2–4GB RAM при локальному розгортанні. На DigitalOcean droplet з 2GB RAM це неприйнятно — не залишається ресурсів для Django, Celery, PostgreSQL та Redis. OpenRouter переносить обчислювальне навантаження на хмарну інфраструктуру.

3. **Sentence-transformers локально.** Модель paraphrase-multilingual-MiniLM-L12-v2 займає ~100MB RAM і генерує embeddings за ~50ms. Локальне виконання забезпечує: відсутність залежності від зовнішнього API для similarity модуля, мінімальну затримку, підтримку 50+ мов (включаючи українську та російську).

4. **Гібридний підхід.** Поєднання зовнішнього API (для важких LLM-задач) та локальних моделей (для embeddings та rule-based аналізу) забезпечує оптимальний баланс між якістю, швидкістю та вартістю. При збої OpenRouter API система продовжує працювати з rhetoric та similarity модулями (graceful degradation).

## 7. Веб-інтерфейс: Django Templates + HTMX

### Розглянуті альтернативи

| Підхід | Переваги | Недоліки |
|--------|----------|----------|
| **Django Templates + HTMX** | Zero-build; серверний рендеринг; мінімальний JS; інтерактивність через HTMX; єдина кодова база з бекендом | Обмежені можливості складного UI; менш інтерактивний ніж SPA |
| React (Create React App / Next.js) | Повноцінний SPA; багата екосистема UI-компонентів | Окрема кодова база; потребує Node.js для build; складніший деплой; CORS-конфігурація |
| Vue.js | Простіший за React; хороша інтеграція з Django | Все ще окрема кодова база та build pipeline |

### Обґрунтування вибору

Django Templates з HTMX обрано для веб-інтерфейсу:

1. **Єдина кодова база.** HTML-шаблони зберігаються у тому ж Django-проєкті, що й API. Не потрібен окремий frontend-репозиторій, build pipeline чи Node.js.

2. **HTMX.** Бібліотека HTMX (~14KB) дозволяє створювати інтерактивні веб-сторінки без написання JavaScript: AJAX-запити через HTML-атрибути, часткове оновлення сторінки, WebSocket-інтеграція. Для нашого інтерфейсу (форма введення тексту, відображення результату, перегляд історії) HTMX повністю достатній.

3. **Швидкість розробки.** Для одного розробника з дедлайном 10 тижнів створення повноцінного React-SPA — надлишкова складність. HTMX дозволяє створити функціональний інтерфейс за 1–2 дні.

## 8. Розгортання та інфраструктура

### 8.1. Сервер: DigitalOcean Droplet

| Характеристика | Значення |
|----------------|---------|
| ОС | Ubuntu 24.04 LTS |
| CPU | 2 vCPU (AMD/Intel) |
| RAM | 2GB |
| Диск | 50GB SSD |
| Регіон | Frankfurt (FRA1) — найближчий до України |
| Вартість | ~$12/міс (наявна підписка) |

DigitalOcean обрано через наявну підписку студента, простоту адміністрування (один droplet з усіма сервісами) та достатню продуктивність для MVP.

### 8.2. Reverse proxy: nginx

nginx обрано як reverse proxy через: високу продуктивність обробки статичних файлів, вбудовану підтримку SSL/TLS (інтеграція з Let's Encrypt через certbot), можливість маршрутизації запитів до Django (proxy_pass), захист від повільних клієнтів та DDoS.

### 8.3. Управління процесами: systemd

systemd використовується для управління всіма сервісами системи: gunicorn (Django WSGI server), celery worker(s), redis-server. Systemd забезпечує автоматичний запуск при завантаженні системи (WantedBy=multi-user.target), автоматичний перезапуск при збоях (Restart=on-failure), логування через journalctl.

### 8.4. WSGI-сервер: Gunicorn

Gunicorn (Green Unicorn) — WSGI HTTP-сервер для Django з підтримкою pre-fork worker model. Конфігурація: 3 workers (рекомендація: 2 × CPU + 1), timeout 120s (для обробки довгих webhook-запитів), bind 127.0.0.1:8003 (доступний лише через nginx).
